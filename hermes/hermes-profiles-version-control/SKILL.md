# Hermes Profiles Version Control

Centralize Hermes agent profiles in a git repository while keeping runtime state local via symlinks.

## Why

- **Version control**: Track profile changes (SOUL.md, config.yaml)
- **Share profiles**: Use same profiles across machines
- **Backup**: Profiles survive system reinstalls
- **Isolation**: Runtime state stays out of git

## Directory Structure

```
~/src/hermes-config/          # Git repo
├── config.yaml               # (symlinked to ~/.hermes/config.yaml)
├── profiles/
│   ├── default/
│   │   ├── config.yaml
│   │   ├── SOUL.md
│   │   └── .env
│   └── developer/
│       ├── config.yaml
│       ├── SOUL.md
│       └── .env
└── .gitignore                # Excludes runtime state

~/.hermes/
├── config.yaml -> ~/src/hermes-config/config.yaml
└── profiles/
    ├── default/              # (copied, not symlinked - for safety)
    └── developer -> ~/src/hermes-config/profiles/developer  # Symlink
```

## Setup

### 1. Create Profile in Hermes

```bash
# Create new profile from current
hermes profile create developer --clone

# Or create empty
hermes profile create my-agent
```

### 2. Move to Version Control

```bash
cd ~/src/hermes-config
mkdir -p profiles/developer

# Copy config files (NOT runtime state like sessions/, logs/, skills/)
cp ~/.hermes/profiles/developer/config.yaml profiles/developer/
cp ~/.hermes/profiles/developer/SOUL.md profiles/developer/
cp ~/.hermes/profiles/developer/.env profiles/developer/

# Add .gitignore for runtime state
cat >> .gitignore << 'EOF'
# Runtime state (generated by Hermes)
profiles/*/sessions/
profiles/*/logs/
profiles/*/cron/
profiles/*/plans/
profiles/*/memories/
profiles/*/skills/
profiles/*/skins/
profiles/*/workspace/
EOF
```

### 3. Create Symlink

```bash
# Remove original
rm -rf ~/.hermes/profiles/developer

# Create symlink
ln -s ~/src/hermes-config/profiles/developer ~/.hermes/profiles/developer
```

### 4. Commit and Push

```bash
git add profiles/developer .gitignore
git commit -m "Add developer profile"
git push origin master
```

## Managing Profiles

### Add Another Profile

```bash
# Create in Hermes
hermes profile create production --clone-from developer

# Move to repo
cp -r ~/.hermes/profiles/production ~/src/hermes-config/profiles/
rm -rf ~/.hermes/profiles/production
ln -s ~/src/hermes-config/profiles/production ~/.hermes/profiles/production
```

### Update Existing Profile

```bash
# Edit in repo
nvim ~/src/hermes-config/profiles/developer/SOUL.md

# Changes apply immediately (symlinked)
hermes profile show developer
```

### Switch Default Profile

```bash
hermes profile use developer  # Persistent
hermes --profile developer    # One-time
```

## What Goes in Git

| File | In Git? | Notes |
|------|---------|-------|
| `config.yaml` | Yes | Model, provider, tools config |
| `SOUL.md` | Yes | Personality, identity |
| `.env` | Yes | API keys (encrypt if sensitive) |
| `sessions/` | No | Runtime chat history |
| `logs/` | No | Runtime logs |
| `skills/` | No | Downloaded/created skills |
| `memory.db` | No | SQLite session history |

## Multi-Machine Workflow

```bash
# Machine A: Create profile
hermes profile create work --clone
cp -r ~/.hermes/profiles/work ~/src/hermes-config/profiles/
ln -sf ~/src/hermes-config/profiles/work ~/.hermes/profiles/work
git add profiles/work && git commit && git push

# Machine B: Pull and use
git pull
ln -s ~/src/hermes-config/profiles/work ~/.hermes/profiles/work
hermes profile use work
```

## Troubleshooting

| Issue | Solution |
|-------|----------|
| Profile not found | Check symlink: `ls -la ~/.hermes/profiles/` |
| Changes not reflected | Ensure editing file in repo, not original |
| Git shows runtime files | Add to .gitignore, remove from tracking |
| Broken symlink | Recreate: `ln -sf ~/src/hermes-config/profiles/X ~/.hermes/profiles/X` |

## Best Practices

1. **Always** add runtime directories to `.gitignore`
2. **Never** commit `memory.db` or session files
3. **Use** `git pull --rebase` before committing profile changes
4. **Test** profile works after symlink before deleting original
5. **Backup** `.env` files if they contain API keys (or use env vars)
